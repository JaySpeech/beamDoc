Python设计模式（四）：行为型模式

发表于 2020-01-04 分类于 编程 阅读次数： Valine：
本文字数： 12k 阅读时长 ≈ 11 分钟
Python设计模式的最后一部分行为型模式，包括啦责任链模式、命令模式、解释器模式、观察者模式、状态模式、策略模式、模版模式

解释器模式、状态模式，由于没大看明白，且目前所浏览的代码没有见过此类型，所以暂且束之高阁。此篇主要放在责任链模式、命令模式、解释器模式、观察者模式、策略模式和模版模式上，内容主要来源于《Mastering Python Design Patterns》中译版。

责任链模式

这个模式并不陌生，公司架构级的Dnn Training Tool中的Feature Extraction就是应用的此种模式，整体类似于链式编程的思想，当如果一节点对某个请求不感兴趣或者不知道如何处理这个请求时，就将请求发送给下个节点。在提取音频特征的时候，每个特征都没有及时处理，而是直接丢给他的下一节点，直到读取到音频文件为止，很值得学习的编程思想，要多看几遍，以下为上述参考书籍的示例代码。

命令模式

命令模式能解一个操作（复制、粘贴）封装为一个对象，这样能提供很多好处。如下所述：

1.可以在任何时候执行一个命令，而并不一定是在命令创建的时候。

2.执行一个命令的客户端代码并不需要知道命令的任何实现细节。

3.可以对命令进行分组，并按一定的顺序执行。

命令模式也可适应于组合多个命令，这有助于实现宏、多级撤销以及事务。一个事务应该：要么成功，这意味着事务中所有操作应该都成功；要么如果至少一个操作失败，则全部失败（回滚操作）。

在做测试开发自动化的时候，命令模式应该是一个不错的选择，或者进一步说对于任何涉及到顺序步骤执行的操作，都可以试试命令模式。

观察者模式

如希望在一个对象的状态改变变化时，能够通知/提醒所有相关者（一个对象或者一组对象），则可以使用观察者模式。

策略模式

策略模式是一种非常重要的模式设计方法，鼓励使用多种算法来解决一个问题。其杀手级的特性是能够在运行时透明地切换算法，如果你有两种算法，其中一种对少量输入效果较好，一个对大量输入效果较好，则可以使用策略模式在运行时基于输入数据决定使用哪种算法，比如排序算法。
